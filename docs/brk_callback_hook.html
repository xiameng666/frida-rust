CRC 对抗：双页面 W^X 状态机

wxshadow 已有完整的 CRC 对抗机制，BRK 带回调方案完全复用：

正常执行路径:
PTE → 影子页(--x) → CPU 取指 → BRK #7 → 内核改 PC → Thunk
CRC 读取路径:
memcmp(0x7200, expected, 20) → LDR 读影子页 → Data Abort (页不可读) → 内核 page_fault handler → 识别为 wxshadow region → 切 PTE 到原始页(r--) → 状态变 ORIG_R → 返回用户态重试 → LDR 成功读到原始字节 ✓ CRC 通过
CRC 后恢复:
CRC 线程继续执行 → 下次 PC 到达同页代码 → Instruction Abort (原始页不可执行) → 内核 page_fault handler → 切 PTE 回影子页(--x) → 状态变 SHADOW_X

状态机转换

                    ┌──────────────────────────────────────┐
                    │         STATE_SHADOW_X               │
                    │     影子页映射, execute-only (--x)     │
                    │     含 BRK #7 在 hook 点              │
                    └───────┬──────────────┬───────────────┘
                            │              │
                  CRC 读取触发        BRK #7 命中
                  Data Abort          Sync Exception
                            │              │
                            ▼              ▼
              ┌─────────────────┐  ┌──────────────────┐
              │  STATE_ORIG_R   │  │  内核 BRK handler │
              │  原始页, r--     │  │  改 PC = thunk   │
              │  CRC 读原始字节  │  │  返回用户态       │
              └────────┬────────┘  └──────────────────┘
                       │
             执行触发 Instruction Abort
                       │
                       ▼
              ┌─────────────────┐
              │ 切回 SHADOW_X   │
              │ 恢复影子页映射   │
              └─────────────────┘
关键: 这个 CRC 对抗机制是 wxshadow 已有的。 看源码第 6-18 行的注释和 page_fault handler (第 870 行)。 BRK 带回调方案不需要额外改动 CRC 对抗逻辑，因为影子页上仍然是 BRK 指令 + execute-only 权限， page_fault handler 照常工作。
对比跳转方案: 如果用 MOVZ/MOVK/BR 跳转写到 r-x 可读页， CRC 直接读到跳转指令，page_fault 不会触发（页可读）， 必须另外 hook CRC 函数。BRK + execute-only 方案天然过 CRC。

多线程安全性

Case 1: 两个线程同时命中同一 hook 点

// BRK #7 是同步异常，每个线程独立触发，独立处理

线程 A: PC=0x7200 → BRK #7 → 内核 handler → 改 A 的 ELR = thunk → ERET
线程 B: PC=0x7200 → BRK #7 → 内核 handler → 改 B 的 ELR = thunk → ERET

// 两个线程各自进入 thunk，各自有独立的栈帧 (SP 不同)
// thunk 的 HookContext 在栈上分配 → 完全线程安全，零共享状态

线程 A: thunk (SP=0xA000) → onEnter → trampoline → onLeave → RET
线程 B: thunk (SP=0xB000) → onEnter → trampoline → onLeave → RET
与当前 BRK 单步方案的关键区别:
当前 wxshadow 的 BRK 命中后要做单步: 切 PTE → 原始页 → 单步 → 切回影子页。 这个切 PTE 操作影响所有线程，因为 PTE 是进程共享的。

带回调方案不需要单步! 内核只改当前线程的 PC/ELR，不动 PTE。 原始指令的执行由 trampoline 负责（已提前重定位）。PTE 始终指向影子页。

Case 2: CRC 线程读取 + 执行线程命中，同时发生

// 这里需要注意: CRC 触发 page_fault 会切 PTE 到原始页

CRC 线程: LDR [0x7200]
  → Data Abort → page_fault handler → 切 PTE 到原始页 (r--)
  → 状态变 ORIG_R

执行线程: PC 到达 0x7200
  → 原始页 r-- 不可执行 → Instruction Abort
  → page_fault handler → 切 PTE 回影子页 (--x)
  → 状态变 SHADOW_X
  → 执行线程重试 → BRK #7 → 内核改 PC → thunk

// 执行线程多了一次 Instruction Abort 的开销
// 但功能正确，不会崩溃，不会绕过 hook

Case 3: 嵌套 Hook (Hook A 的回调中调用了 Hook B)

hook A: android_dlopen_ext (0x7200)
hook B: __dl_open (0x9400)

调用链:
  app 调 dlopen_ext → BRK@0x7200 → 内核 → thunk_A
    thunk_A: onEnter_A → trampoline_A
      trampoline_A: 执行原始第1条 → B 0x7204 → 原函数体
        原函数内部调 __dl_open → BRK@0x9400 → 内核 → thunk_B
          thunk_B: onEnter_B → trampoline_B → 原函数体 → RET
        thunk_B: onLeave_B → RET (回到 dlopen_ext 内部)
      dlopen_ext RET → 回到 thunk_A
    thunk_A: onLeave_A → RET (回到 app)

// 每个 thunk 有独立栈帧，嵌套自然支持
// 不同 hook 在不同页面，BRK 独立触发

wxshadow 改造方案

核心思路：最小改动

BRK 带回调方案复用 wxshadow 95% 的现有代码。只需改 BRK handler 里的 3 行核心逻辑。

改动 1: bp_entry 新增 thunk_addr 字段

struct bp_entry {
    uint64_t       addr;
    uint8_t        active;
    uint8_t        has_thunk;          // 新增: 是否有 thunk 回调
    uint8_t        _pad[6];
    struct reg_mod mods[MAX_REG_MOD];
    uint32_t       mod_count;
    uint64_t       thunk_addr;         // 新增: 用户态 thunk 地址
};

改动 2: 新增 prctl 命令 SET_BP_THUNK

// 新增命令码
#define WXSHADOW_PRCTL_SET_BP_THUNK  0x57585804

// 用法: prctl(0x57585804, pid, hook_addr, thunk_addr)
// 效果: 在 hook_addr 设置 BRK #7，命中时内核将 PC 改为 thunk_addr

改动 3: BRK handler 增加 thunk 分支 (核心 ~10 行)

// wxshadow_brk_handler() 中, 找到 bp 之后:

struct bp_entry *bp = wxshadow_find_bp(pi, pc);

if (bp && bp->has_thunk && bp->thunk_addr) {
    /* ===== 带回调模式: 改 PC 跳到用户态 thunk ===== */
    /* 不做单步，不切PTE，不改寄存器 */
    /* 只需要一行核心操作: */
    regs->pc = bp->thunk_addr;

    /* 日志 */
    pr_info("wxshadow: BRK thunk redirect: pc=%lx -> thunk=%lx\n",
            pc, bp->thunk_addr);

    wxshadow_put_region(r);
    kfn_mmput(mm);
    return 0;  /* 处理完毕 */
}

/* 否则走原来的 SET_REG + 单步 逻辑 */
// ... 现有代码不动 ...

改动 4: prctl handler 增加 SET_BP_THUNK 分支

if ((uint32_t)option == WXSHADOW_PRCTL_SET_BP_THUNK) {
    /* arg2 = hook_addr, arg3 = thunk_addr */
    mm = resolve_target_mm(pid);
    if (!mm) { ... }

    /* 复用现有 set_bp 逻辑创建影子页 + 写 BRK */
    ret = wxshadow_do_set_bp(mm, addr);
    if (ret == 0) {
        /* 设置 thunk 地址 */
        struct bp_entry *bp = find_bp_by_addr(mm, addr);
        if (bp) {
            bp->has_thunk = 1;
            bp->thunk_addr = arg3;
        }
    }
    kfn_mmput(mm);
    fargs->skip_origin = 1;
    fargs->ret = ret;
    return;
}

不需要改动的部分 (全部复用)

✅ 影子页创建 / PTE 切换 / TLB 刷新 — wxshadow_do_set_bp() 原样复用
✅ CRC 对抗的 page_fault handler — 完全不动，BRK 页面的 execute-only 权限照常
✅ 区域管理 (region_list) — 不动
✅ 进程退出清理 (exit_mmap_before) — 不动
✅ DEL_BP — 原样复用，删除时清 thunk_addr 即可
✅ BRK handler 入口过滤 (BRK #7 + EL0 检查) — 不动

用户态 (hook_engine.c) 改动

// stealth 模式改为用 SET_BP_THUNK 代替 wxshadow_patch

if (stealth) {
    /* 1. 生成 trampoline: 只重定位 1 条指令 + 跳回 hook_addr+4 */
    relocated = hook_relocate_instructions(target, trampoline, 4); // 只要 4 字节!
    hook_write_jump(trampoline + relocated, target + 4);

    /* 2. 生成 thunk (复用 generate_attach_thunk) */
    thunk = generate_attach_thunk(entry, on_enter, on_leave, user_data, &thunk_size);

    /* 3. 通知内核: 在 target 设 BRK，命中时跳 thunk */
    prctl(0x57585804, 0, target, thunk, 0);
}

完整调用序列

安装 Hook:
用户态 → 生成 trampoline (重定位1条指令) → 生成 thunk (保存寄存器+回调+调原函数) → prctl(0x57585804, 0, hook_addr, thunk_addr) → 内核: 创建影子页 + 写 BRK #7 + 记录 thunk_addr + 切 PTE
运行时命中:
CPU 取指 → BRK #7 → 内核: regs->pc = thunk_addr → ERET → thunk: onEnter → trampoline (原指令+跳回) → 原函数 → RET 回 thunk → onLeave → RET 回调用者
CRC 检测:
memcmp 读 hook_addr → Data Abort (execute-only) → page_fault handler → 切到原始页 → 读到原始字节 ✓
