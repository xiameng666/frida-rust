① 什么是 mm_struct？

在 Linux 内核中，每个用户态进程都有一个唯一的 mm_struct（Memory Management Structure，内存管理描述符）。 它是内核用来描述、管理该进程全部虚拟地址空间的核心数据结构。

一句话理解： mm_struct 就是内核眼中一个进程的"地图"——记录了这个进程占用了哪些虚拟内存、 页表在哪里、ASID 是多少等一切内存相关信息。

task_struct → mm_struct 的关系

每个线程有一个 task_struct，主线程的 task_struct.mm 指向进程的 mm_struct。内核线程（如 swapper/0）没有用户地址空间，所以 mm == NULL，只有 active_mm 有值。

⚠️ 内核线程特殊情况：
swapper/0（init_task，PID=0）是内核线程，mm == NULL， 但 active_mm 指向 &init_mm（一个非空的全局变量）。 这就是为什么 KP 框架能检测到 active_mm_offset 却检测不到 mm_offset。

② mm_struct 关键字段详解

点击各字段查看详细说明

// include/linux/mm_types.h struct mm_struct { struct { struct vm_area_struct *mmap; /* VMA 链表头 — 所有内存映射 */ struct rb_root mm_rb; /* VMA 红黑树（快速查找） */ pgd_t *pgd; /* ⭐ 页表根 — 虚拟→物理地址翻译起点 */ atomic_t mm_users; /* 引用此 mm 的线程数 */ atomic_t mm_count; /* mm 自身引用计数 */ int map_count; /* VMA 总数量 */ unsigned long start_code, end_code; /* 代码段范围 */ unsigned long start_data, end_data; /* 数据段范围 */ unsigned long start_brk, brk; /* 堆的起止地址 */ unsigned long start_stack; /* 栈底地址 */ /* ... 还有几十个字段 ... */ }; mm_context_t context; /* ⭐ 架构相关上下文 (ARM64 = ASID 等) */ }; // arch/arm64/include/asm/mmu.h typedef struct { atomic64_t id; /* ⭐ ASID (地址空间 ID) — 区分不同进程的 TLB 条目 */ void *vdso; /* vDSO 库映射地址 */ spinlock_t lock; /* 保护 id 字段的自旋锁 */ } mm_context_t;

pgd — 页表根，最重要的字段

pgd 是页全局目录的虚拟地址指针。ARM64 有 4 级页表：

pgd → PUD → PMD → PTE → 物理页（4KB）

CPU 执行用户态代码时，先把 pgd 的物理地址写入 TTBR0_EL1 寄存器，然后 MMU 自动用它翻译所有虚拟地址。

mmap + mm_rb — VMA 管理

进程的每一段内存（.text、.so、stack、heap…）都是一个 vm_area_struct (VMA)。mmap 是链表头， mm_rb 是红黑树（O(log n) 查找）。 wxshadow 的 find_vma() 调用就是在查这棵树。

context.id — ASID（地址空间 ID）

标准 Linux 存储格式：
context.id.counter = ASID | (generation << ASID_BITS)
低 16 位 = 硬件 ASID，高位 = 代次计数器（ASID 轮回次数）

MTK 定制（你的设备）：
context.id.counter = 完整 TTBR0 值 = phys_pgd | (ASID << 48)
高 16 位 = ASID，低 48 位 = 物理页表地址 + CnP 位

③ ASID & TTBR0_EL1 — 为什么进程需要 ID？

每次进程切换，CPU 的 TLB（转换后备缓冲区）里还存着旧进程的地址翻译结果。 如果不清空 TLB，新进程可能读到错误的物理地址 → 崩溃或安全漏洞。

清空整个 TLB 代价很高。ARM64 引入了 ASID （Address Space ID），让不同进程的 TLB 条目共存， 只需按 ASID 精准无效化，不需要全清。

TTBR0_EL1 寄存器结构（来自你的日志 0x0e1800012e50d001）

0x0e18 = 3608
bits[47:1]
物理 PGD 基地址
0x00012e50d000
bit[0]

1
CPU 切换进程时做了什么：
MSR TTBR0_EL1, X0 — 将新进程的 phys_pgd | (ASID<<48) 写入寄存器
这一条指令让 MMU 立刻切换到新进程的页表，TLB 中带旧 ASID 的条目自动被忽略

MTK 内核的定制：context.id 存储完整 TTBR0

标准内核的 context.id 只存 ASID | generation， 而 MTK 为了加速上下文切换，直接把完整的 TTBR0 值预计算并缓存在 context.id.counter 里：

ASID = 0x0e18
← 我们需要这里
bits[47:1]
物理 PGD 地址
bits[16:31] = 0x2e50 ← 原版二进制找的
bit[0]
CnP
1
✅ 修复后的扫描逻辑：检查 bits[48:63] 是否等于 ASID = ttbr0 >> 48。找到后，读 ASID 时用 (val >> 48) & 0xFFFF 而不是低 16 位。

④ 为什么要"扫描"偏移量？

内核编译时，结构体字段的偏移量会因配置（CONFIG_*）、内核版本、 厂商补丁而变化。wxshadow 不能硬编码这些数值，必须在运行时动态检测。

为什么 KP 框架能检测 active_mm_offset 却检测不到 mm_offset？

KP 的检测原理：在 init_task（swapper/0）的内存里找一个特定的非空指针。
— active_mm 指向 &init_mm（非空）→ 可以找到 ✅
— mm = NULL（swapper 是内核线程，没有用户内存）→ 找不到 ❌

mm_offset 检测方法（我们的修复）

ARM64 Linux 中，task_struct 里 mm 和 active_mm 相邻：

/* 典型布局（ARM64 Linux 5.10）*/ offset+0: struct mm_struct *mm; ← 用户进程有mm，内核线程=NULL offset+8: struct mm_struct *active_mm; ← 始终非空，指向当前使用的mm // 因此: mm_offset = active_mm_offset - 8 // 验证: *(init_task + candidate) == 0 (swapper的mm=NULL) // AND: *(init_task + active_mm_off) >> 48 == 0xFFFF (有效内核指针)

⑤ wxshadow 在哪里用到 mm_struct？

wxshadow 的无痕 hook 本质是影子页技术： 用两个物理页（原始页 + 影子页）通过修改页表（PTE）在同一虚拟地址上切换。 这一切都需要通过 mm_struct 来操作。

🗺️
查找 VMA
设置 hook 时，用 find_vma(mm, addr) 确认地址属于哪个 VMA，获取 VMA 的起止范围。
kfn_find_vma(mm, addr)
📄
遍历页表
通过 mm.pgd 字段找到页表根，然后 4 级遍历（PGD→PUD→PMD→PTE）找到目标地址的 PTE。
mm + pgd_offset → *pte
🔄
切换物理页
BRK 命中时，将 PTE 从影子页（含 BRK 指令）切换到原始页（正常代码）， 单步执行完再切回去。
*pte = make_pte(orig_pfn, flags)
💨
TLB 刷新（最后回退）
修改 PTE 后必须刷新 TLB。优先用内核函数，实在没有时 用 ASID 精准刷新（tlbi vale1is）。
tlbi vale1is, addr|(asid<<48)

完整数据流

// 1. 从当前 task 获取 mm（用 mm_offset） void *mm = kfn_get_task_mm(current); // 2. 用 pgd_offset 读页表根 uint64_t pgd = *(uint64_t*)(mm + pgd_offset); // 3. 4级页表遍历找到 PTE uint64_t *pte = get_user_pte(mm, target_addr); // 4. 切换到影子页（BRK 指令在里面） *pte = make_pte(shadow_pfn, 0); // 5. 刷新 TLB（优先调用内核函数） flush_tlb_page(vma, addr); // 或回退: tlbi vale1is, addr | (asid << 48)
关键结论： mm_offset 是整个 wxshadow 的基础，没有它就无法找到页表（pgd）， 无法修改 PTE，无法切换影子页，无痕 hook 完全失效。 context.id 只是 TLB 刷新的最后兜底，优先级低得多。
